We are living in an era, where every company is going through a digital transformation; no matter how big the company is, it must adapt to the modern world. And that might be the biggest change in their history. Therefore, industry quality code is a standard that the company produces and upon which they build their business. If it does not work, it will cost a lot of money because then their business does not work. It is crucial that the code that the company utilizes to run its business is of the highest possible quality. Everything else is of lesser priority.

Software development is mostly teamwork, but not any ordinary type of teamwork. Most of the time each team member works alone and the process of software development itself could be seen as the teamwork aspect of it all: every team member follows strict process rules and the other team members ensure that everybody is following it. We are a strong believer that quality is built within teams, and teams need processes to make sure that they work efficiently and produce high quality code. For example, when a company manufactures a product that consumers buy and use, what does a consumer actually pay for? Besides raw materials, they pay mostly for the process that manufactured the product. The same goes for code; code in itself is not that valuable, the process that can produce the code is the most valuable.

When companies produce code on which they run their businesses, the most critical process is a software development process that is used to produce the code. By following a strict process it is possible to impose high quality code. Next, let’s take a look at some points that can be used as quality measurements:

- **Documentation and specification coverage**  
  Everything starts with documentation and specification of business logic that is based on business use cases. It is crucial to write clear business logic specifications so that developers can implement accordingly and to have a clear understanding of all functionalities within the team deliverables. Keeping documentation updated is also important for new developers and when revisiting business logic when business use cases change.

- **Static code analysis coverage**  
  To spot problems early, every line that a developer writes should be analyzed to ensure it conforms to coding guidelines and does not produce code smells in the code-base.

- **Unit test code coverage**  
  The process of writing code should contain automated tests that execute every line of code to make sure that nothing breaks when code is changed and that the functionality still works as expected. Good unit tests can be seen as implementation documentation on a functional or atomic level.

- **Code review coverage**  
  Preferably all team members should review everybody’s code to make sure that it looks good, has good unit tests testing every line and that it follows other development rules that are set.

- **Business case test coverage**  
  To make sure that every line of code produces real business value, business logic should be mapped into the code and there should be tests for each behavior in the business logic. These tests should be automated so that manual work is minimized, and tests can be executed before releasing any of the newly developed code.

- **Experiential testing coverage**  
  It is also important to do black box, penetration and non-functional testing that tries to break the code.

- **Number of reported bugs**  
  When code is released, and it is running in production, there will be bugs. The number of bugs should be minimized because every bug in production is critical.

- **Team confidence level**  
  How confident is the team of the work they do? If some team members are not confident of their own work, the root cause of non-confidence must be figured out and resolved.

- **Team velocity**  
  How fast can the team produce code? If the team works efficiently, it should be able to produce high-quality code fast.

- **Expertise level of team members**  
  In the end, it is people who write the code. They must know their craftsmanship. It is important that the team has a balanced competence pool and preferably all members can perform all duties within a team.

- **End-user happiness**  
  Because code is written for other people to consume, it is important that they enjoy using the end result. If they don’t like it, the root cause must be identified adjusted accordingly.

All these points above should be included in the software development process. It is how quality is built within a team that ultimately produces high quality industry code. Remember that code itself is not that valuable because it often has a short life cycle and changes often over time. The ability to produce high quality industry code is what is the most valuable aspect for a company that heavily relies on the code they produce.
