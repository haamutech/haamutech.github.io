Some interesting solved problems

High amounts of technical debt

2021 – 2020
SATO, real estate company, revenue 220+ million euros

SATO is reshaping its business to be competitive in future in highly digitalizing market. All of the core business services and processes are transformed to software and it creates a constant pressure to develope new software solutions. To do this effectively, SATO needs to have very strict software development process. SATO was lacking an agile software development process and because of that, each piece of code that was written accumulated technical debt. I formed a software development process using agile and lean principles to cut off producing technical debt. The process includes CI pipelines that check every commit and enforce code quality standards, set clear goals to meetings and remove ones that do not bring value, create guidelines for business on how to write specification, introduce unit testing to codebase and run tests automatically. The end result was that developer happiness started to grow and the codebase started to heal.

Poor and risky maintainability

2020 – 2018
ABB, manufacturing company, revenue 27+ billion dollars

ABB's IoT Edge solution started as a proof-of-concept (PoC) which grew into product. The PoC was written in Node.js, but because it originated from PoC and was never rewritten, it was lacking the critical software engineering practices. I started to rewrite the solution to C# and at the same time coming up with Behavior-Driven Development (BDD) system that we used as a framework for our specifications. With BDD specs, we were able to identify all the use cases existing PoC had and rewrite those in C# with Test-Driven Development (TDD). The finished product had all use cases specified in BDD, test coverage of 100% and it was much easier to maintain and test before deploying it to sites.

Make API response times less than 1 second

2018 – 2017
Caverion Oyj, industrial service company, revenue 2+ billion euros

The customer utilized microservice architecture where the external API facade was created via Azure API Management that called internal Node.js APIs running in Rancher Docker cluster. We had around 10 API services, and it is typical that APIs can call each other internally. The customer's requirement was to get all API requests to complete in less than 1 second. I solved it by optimizing APIs to use Node.js streams where appropriate and built an indexing API which used Azure Search for storing slow SAP HANA master data queries. The measured API response times were capped to 500ms.
